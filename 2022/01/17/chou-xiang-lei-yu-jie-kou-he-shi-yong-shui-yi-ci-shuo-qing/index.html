抽象类与接口——何时用谁，一次说清

发布于 2022-01-17 分类：技术, 2022, Java基础 👁

前言

“抽象类和接口有什么区别？”——这是面试必考题，但实际开发中真正让人纠结的是：这个场景到底该用抽象类还是接口？ 2022年我通过大量实践，总结出了一套简单实用的判断标准。

核心区别一览

维度 抽象类 接口
关键字 abstract class interface
实例化 不能 不能
构造方法 ✅ 可以有 ❌ 不能有
成员变量 任意类型 public static final 常量
方法 抽象+具体 抽象+default+static
访问修饰符 任意 public（默认）
多继承 单继承 多实现
设计语义 is-a（是什么） can-do（能做什么）

JDK8/9/11 接口的变化

接口越来越”像”抽象类了，但本质区别不变：接口没有状态（成员变量），抽象类有。

判断标准：三问法

第一问：有没有公共状态？

  • 有公共字段/状态 → 抽象类
  • 只定义行为，无状态 → 接口

第二问：有没有模板逻辑？

  • 有公共流程骨架，子类填充细节 → 抽象类
  • 各实现独立，无公共流程 → 接口

第三问：需不需要多继承？

  • 一个类需要具备多种能力 → 接口
  • 只需要一种身份 → 抽象类或接口均可

项目中的典型选择

场景1：实体类基类 → 抽象类

原因：有公共状态（id、时间字段），子类自动继承。

场景2：策略/处理器 → 接口

原因：无公共状态，只定义行为契约，支持多实现。

场景3：模板流程 → 抽象类 + 接口配合

场景4：能力标记 → 接口

常见误区

1. “接口优于抽象类”的教条

很多文章说”优先用接口”，但这是有前提的。如果你的场景有公共状态和模板逻辑，强行用接口会导致：

  • default方法泛滥
  • 常量接口反模式
  • 逻辑分散在多个default方法中，难以维护

2. 常量接口反模式

3. 接口承载过多逻辑

总结

抽象类和接口的选择，核心是理解它们的设计语义：抽象类是”is-a”，定义你是什么；接口是”can-do”，定义你能做什么。2022年我总结的实用原则是：

  1. 有公共状态 → 抽象类
  2. 有模板流程 → 抽象类
  3. 只定义行为 → 接口
  4. 需要多继承 → 接口
  5. 不确定 → 先用接口，后续需要时再加抽象类
