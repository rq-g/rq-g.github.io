重载与重写——一字之差，天壤之别

发布于 2022-01-14 分类：技术, 2022, Java基础 👁

前言

重载（Overload）和重写（Override）是Java中最容易混淆的两个概念。刚入行的时候，我只知道”重载是方法名相同参数不同，重写是子类覆盖父类方法”，但真正在项目中遇到的问题远比这复杂：重载方法的匹配顺序是什么？重写时返回值可以不同吗？静态方法能被重写吗？

重载（Overload）

定义

在同一个类中，方法名相同，参数列表不同（参数类型、个数、顺序不同），与返回值无关。

重载的匹配规则

当调用重载方法时，编译器按以下顺序匹配：

  1. 精确匹配 → 参数类型完全一致
  2. 基本类型自动转换 → int → long → float → double
  3. 自动装箱 → int → Integer
  4. 可变参数 → int… → 匹配

重载的坑

重写（Override）

定义

子类重新定义父类中签名相同的方法，运行时根据实际对象类型调用对应版本。

重写的规则（@Override 约束）

规则 说明
方法签名 必须与父类完全一致
返回值 相同，或是父类返回值的子类（协变返回）
访问权限 不能比父类更严格
异常 不能抛出比父类更多的检查异常
static方法 不能被重写（属于类，不属于实例）
final方法 不能被重写
private方法 不能被重写（子类不可见）

协变返回类型

重载 vs 重写 对比

维度 重载（Overload） 重写（Override）
发生位置 同一个类 父子类之间
方法名 相同 相同
参数列表 必须不同 必须相同
返回值 无关 相同或协变
访问权限 无限制 不能更严格
异常 无限制 不能更多
绑定时机 编译期（静态绑定） 运行期（动态绑定）
注解 无 @Override

项目中的实战场景

重载：工具类方法

重写：业务策略

总结

重载和重写虽然名字相似，但本质完全不同：重载是编译期静态行为，解决”同一个操作适配不同参数”的问题；重写是运行期动态行为，解决”同一个接口不同实现”的问题。2022年踩过的坑告诉我：永远用 @Override 注解标注重写方法，让编译器帮你检查，避免”以为重写了实际是重载”的低级错误。
