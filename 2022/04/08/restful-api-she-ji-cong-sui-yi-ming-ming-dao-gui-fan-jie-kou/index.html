RESTful API设计——从随意命名到规范接口

发布于 2022-04-08 分类：技术, 2022, SpringBoot 👁

前言

刚写接口的时候，URL命名随心所欲：/getUser、/deleteOrder、/queryList，各种风格混搭。直到和前端联调时，接口命名不统一、状态码不一致、返回格式混乱，联调效率极低。后来统一了RESTful规范，前后端协作顺畅了很多。

什么是RESTful？

REST（Representational State Transfer）是一种架构风格，核心思想：用URL定位资源，用HTTP方法描述操作。

核心原则

  1. URL表示资源：名词，不用动词
  2. HTTP方法表示操作：GET查询、POST新增、PUT修改、DELETE删除
  3. 状态码语义化：200成功、400客户端错误、404不存在、500服务端错误
  4. 返回格式统一：统一的JSON结构

HTTP方法语义

方法 语义 幂等性 安全性
GET 查询 幂等 安全
POST 新增 非幂等 不安全
PUT 全量修改 幂等 不安全
PATCH 部分修改 幂等 不安全
DELETE 删除 幂等 不安全

幂等性：多次调用结果一致。GET查询多次结果一样，PUT全量替换多次结果一样，POST每次新增一条记录所以非幂等。

最佳实践

  1. URL用名词复数：/users而非/user
  2. HTTP方法语义化：GET/POST/PUT/DELETE
  3. 返回格式统一：code + msg + data
  4. 参数校验用@Valid：不要在Controller写if-else校验
  5. DTO分层：请求用CreateDTO/UpdateDTO，响应用VO
  6. API文档同步维护：Swagger注解与代码一起更新

总结

RESTful API设计不是教条，而是一种约定。2022年我从”随意写接口”到”规范设计API”，最大的收获是：好的API设计让前后端协作效率翻倍。URL表示资源、HTTP方法表示操作、返回格式统一、状态码语义化——这四条原则足够覆盖90%的接口设计场景。
