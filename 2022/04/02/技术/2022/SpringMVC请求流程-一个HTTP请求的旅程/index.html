SpringMVC请求流程——一个HTTP请求的旅程

发布于 2022-04-02 分类：技术, 2022, SpringBoot 👁

前言

写了一段时间的Controller，@RequestMapping、@RequestBody、@ResponseBody用得很溜，但一个HTTP请求从浏览器发出到Controller方法执行，中间经历了什么？这个问题困扰了我很久，直到梳理了SpringMVC的完整请求流程。

SpringMVC核心组件

组件 作用
DispatcherServlet 前端控制器，统一入口
HandlerMapping 处理器映射，找到对应的Controller方法
HandlerAdapter 处理器适配器，调用Controller方法
HandlerInterceptor 拦截器，在Controller前后执行逻辑
Controller 处理器，业务逻辑
ViewResolver 视图解析器（前后端分离后很少用）
ModelAndView 模型和视图

过滤器Filter vs 拦截器Interceptor

特性 Filter Interceptor
规范 Servlet规范 Spring规范
作用范围 所有请求 仅DispatcherServlet处理的请求
执行时机 在DispatcherServlet前后 在Handler前后
获取Bean 不方便 方便注入
适用场景 编码、跨域、压缩 权限、日志、事务

最佳实践

  1. 理解请求流程：遇到问题知道在哪一层排查
  2. Filter处理通用逻辑：编码、跨域、XSS过滤
  3. Interceptor处理业务逻辑：权限、日志、上下文
  4. 全局异常处理：统一异常返回格式
  5. 不要在Filter/Interceptor中写复杂业务逻辑

总结

理解SpringMVC的请求流程，是从”会用”到”理解”的关键一步。一个HTTP请求的旅程：DispatcherServlet → HandlerMapping → Interceptor.preHandle → HandlerAdapter → Controller → Interceptor.postHandle → 响应。知道每一层在做什么，遇到问题就能精准定位，而不是瞎猜。
