SpringMVC拦截器与过滤器——区别、执行顺序与应用场景

发布于 2026-02-08 分类：技术, 2026, SpringBoot, 面试题 👁

前言

拦截器和过滤器是SpringMVC中处理请求的两个重要组件，面试时经常被问到它们的区别和应用场景。022年我在项目中需要实现接口权限校验，当时纠结该用Interceptor还是Filter，后来搞清楚了它们的执行顺序和适用边界。

一、拦截器 vs 过滤。

1. 核心区别

维度 过滤器（Filter 拦截器（Interceptor
所属技术 Servlet规范 SpringMVC框架
作用范围 所有HTTP请求 仅SpringMVC请求
执行时机 请求进入容器后，Servlet之前 DispatcherServlet之后，Controller之前
粒度 粗粒度（URL级别。 细粒度（方法级别。
访问范围 只能访问request/response 可访问Spring容器、Handler、ModelAndView
配置方式 web.xml或@WebFilter @Configuration配置。
继承关系 实现Filter接口 实现HandlerInterceptor接口

二、执行顺序详情

1. 过滤器顺序

过滤器的执行顺序。@Order 注解决setOrder() 方法控制，数字越小越先执行。

2. 拦截器顺序

拦截器的顺序由注册顺序决定，先注册的先执行（preHandle），后注册的先执行（postHandle/afterCompletion）。

注意：如果某个Interceptor的preHandle返回false，后续的Interceptor和Controller都不会执行，且之前执行过的Interceptor的afterCompletion也不会执行。

四、典型面试题

1. 过滤器和拦截器的区别是什么？

答案 - 过滤器属于Servlet规范，拦截器属于SpringMVC框架

  • 过滤器在Servlet之前执行，拦截器在DispatcherServlet之后执行
  • 过滤器只能访问request/response，拦截器可以访问Spring容器和Handler
  • 过滤器粒度粗（URL级别），拦截器粒度细（方法级别）

2. 如果多个拦截器，执行顺序是怎样的？

答案 - preHandle：按注册顺序执行

  • postHandle：按注册顺序逆序执行
  • afterCompletion：按注册顺序逆序执行

3. 如何让过滤器访问Spring容器中的Bean

答案 ```java
// 方式1：实现ApplicationContextAware
public class MyFilter implements Filter, ApplicationContextAware {

private ApplicationContext context;

@Override
public void setApplicationContext(ApplicationContext applicationContext) {
    this.context = applicationContext;
}

@Override
public void doFilter(...) {
    MyService service = context.getBean(MyService.class);
    service.doSomething();
}

}

// 方式2：使用@Autowired（需要在配置类中注册。@Component
public class MyFilter implements Filter {
@Autowired
private MyService service;
}

五、最佳实现

场景 推荐方案
统一编码设置 Filter
CORS跨域处理 Filter
权限校验 Interceptor
请求日志记录 Interceptor
接口限流 Interceptor（简单）/ Filter（复杂）
静态资源处理 Filter

总结

过滤器和拦截器虽然都能拦截请求，但职责不同：

  • *过滤。：处理所有HTTP请求，关注请求层面的处理（编码、跨域等。- *拦截。：处理SpringMVC请求，关注业务层面的处理（权限、日志等。
    记住这个原则。需要访问Spring容器或细粒度控制用Interceptor，处理所有HTTP请求用Filter*
