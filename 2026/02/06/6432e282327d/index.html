三、缓存三大问题（项目实战重点）

1. 缓存穿透（智慧社区项目实战）

项目背景 - 5000+设备，但有些设备ID不存在

  • 用户查询不存在的设备时，直接穿透到MySQL
  • 高峰期大量不存在设备的查询，导致MySQL压力过大

项目数据 - 穿透请求占比：5%

  • MySQL压力增加30%
  • 解决方案：缓存空值

*面试追问1：布隆过滤器能解决缓存穿透吗。

2. 缓存击穿（智慧社区项目实战）

项目背景 - 热点设备数据（如小区门禁）过期

  • 大量请求同时查询，直接打到MySQL
  • MySQL瞬间压力暴增

项目数据 - 门禁设备数量：200+

  • 查询QPS：1000+/秒（高峰期）
  • 解决方案：互斥锁 + 热点数据永不过期

面试追问2：如何判断热点数据？

3. 缓存雪崩（智慧社区项目实战）

项目背景 - 凌晨0点，大量设备数据同时过期

  • 5000+设备同时上线，缓存批量失效
  • MySQL瞬间压力暴增，系统崩溃
    项目数据 - 同时过期Key数量：5000+
  • MySQL QPS：10万（正常5000）
  • 解决方案：随机过期时间 + 热点数据预热

面试追问3：缓存雪崩和缓存击穿有什么区别？

四、Redis分布式锁（项目实战重点）

1. 分布式锁实现（智慧社区项目实战）

项目背景 - 设备控制需要互斥（同一时间只能执行一个控制命令）

  • 库存扣减需要原子性（防止超卖。- 分布式环境下，需要分布式。
    项目数据 - 设备控制QPS 和 000. - 锁竞争比例：10%
  • 解决方案：Redis分布式锁（Redisson

Redisson原理。

  1. 看门狗机制（Watchdog - 锁默认过期时间：30. - 看门狗每10秒检查一、 - 如果锁还在持有，自动续期30. - 业务执行完成前，锁不会过期
  2. 续期逻辑
    • 续期时间 = 锁过期时序/ 3 = 10. - 如果业务执行时间超过10秒，自动续期
    • 续期最多执行0次（30分钟。
  3. 与普通Redis锁的区别
    • 普通Redis锁：锁过期后直接释放
    • Redisson锁：锁过期前自动续期

踩坑1：锁过期时间设置不合。→错误：锁过期时间设置。0秒，业务执行需。0.→正确：根据业务执行时间调整锁过期时间

踩坑2：没有正确释放锁
→错误：直接在finally中删除锁
→正确：先检查锁是否由当前线程持有，再删。
踩坑3：高并发下锁竞争激。→错误：所有请求竞争同一把锁
→正确：按业务拆分锁（如按设备ID
踩坑4：分布式锁无法替代数据库事务
→错误：认为加了分布式锁就可以不用事务
→正确：分布式。+ 数据库事务一起使。```

2. 核心经验总结

Q1：Redis为什么这么快。A：单线程 + IO多路复用 + 内存存储

Q2：Redis和Memcache有什么区别？
A：Redis支持更多数据结构、有持久化、支持事务
Q3：Redis如何做高可用。A：主从复。+ Sentinel哨兵 + Cluster集群

Q4：Redis Cluster如何处理数据分片。A：Redis使用16384个槽位，Key通过CRC16哈希后取模分析
Q5：Redis Cluster如何处理节点故障。A：主从自动切。+ 故障转移 + 集群自愈
