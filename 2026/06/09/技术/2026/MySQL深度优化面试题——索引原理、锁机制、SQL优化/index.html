前言

MySQL优化是高级Java开发必须掌握的技能，能写出高效SQL、设计合理索引的开发者，才能应对高并发场景。2023年我参与过多次SQL优化项目，从慢SQL优化到索引重构，积累了不少实战经验，这篇文章整理了MySQL最核心的优化面试题。

一、InnoDB B+树索引原理

1. B+树结构

2. B+树 vs B树

维度 B树 B+树
数据存储 叶子节点和非叶子节点都存储数据 仅叶子节点存储数据
索引大小 大（非叶子节点存数据） 小（非叶子节点只存键）
范围查询 需要回溯 叶子节点链表直接遍历
磁盘IO 多（树高更高） 少（树高更低）
适用场景 随机查询为主 范围查询为主

3. B+树 vs 二叉树

维度 二叉树 B+树
树高 O(n)（最坏情况） O(logn)
磁盘IO 多（树高太高） 少（树高可控）
查询效率 不稳定 稳定
适用场景 内存数据 磁盘数据

1. 核心区别

维度 聚簇索引 非聚簇索引
数据存储 叶子节点存储完整数据 叶子节点存储主键
数量限制 一个表只能有一个 可以有多个
索引结构 数据即索引 索引和数据分离
查询效率 最高（无需回表） 次高（需要回表）

3. 索引失效场景汇总

场景 示例 原因
跳过最左前缀 WHERE b = 2（索引a,b,c） 不符合最左前缀原则
索引列参与计算 WHERE a + 1 = 10 无法使用索引
索引列使用函数 WHERE SUBSTRING(name, 1, 2) = '张' 无法使用索引
模糊查询开头 WHERE name LIKE '%张' 无法使用索引
OR条件两端无索引 WHERE a = 1 OR b = 2 无法使用索引
类型不匹配 WHERE id = '1'（id是int） 隐式转换，无法使用索引
NOT IN/NOT EXISTS WHERE id NOT IN (1, 2, 3) 优化器可能选择全表扫描
IS NULL/IS NOT NULL WHERE name IS NULL 某些场景下无法使用索引
隐式字符编码转换 表A字符集utf8，表B字符集utf8mb4 无法使用索引

1. 锁的分类

维度 类型 说明

粒度 间隙锁 锁住索引间隙
性质 共享锁（S锁） 读锁，多个事务可共享
性质 排他锁（X锁） 写锁，独占

2. 行锁 vs 表锁

维度 行锁 表锁
粒度 细 粗
并发度 高 低
开销 大 小

适用场景 高并发读写 批量操作
