分布式核心难题面试题——分布式事务、分布式ID、分布式锁

发布于 2026-06-07 分类：技术, 2026, 分布式, 面试题 👁

前言

分布式系统是大型互联网公司的基础设施，分布式事务、分布式ID、分布式锁是分布式系统中最核心的三个难题。2024年我参与过多个分布式项目的架构设计，遇到过分布式事务一致性、ID生成冲突等问题，这篇文章整理了分布式最核心的面试题。

一、分布式事务

1. 分布式事务产生的原因

2. CAP理论

特性 说明
Consistency 一致性：所有节点数据一致
Availability 可用性：每个请求都能得到响应
Partition tolerance 分区容错性：网络分区时系统仍能工作

3. BASE理论

特性 说明
Basic Availability 基本可用：允许降级
Soft state 软状态：状态允许短暂不一致
Eventual consistency 最终一致性：最终数据会一致

4. Seata三种模式对比

维度 AT模式 TCC模式 SAGA模式
实现方式 自动回滚（基于undo log） 手动编写try/confirm/cancel 状态机驱动
侵入性 低（无侵入） 高（需要改造业务） 中（需要定义状态）
适用场景 大多数场景 强一致性要求高 长事务、多服务调用
性能 高 中 低
复杂度 低 高 中

二、分布式ID生成方案

1. 常见方案对比

方案 优点 缺点 适用场景
数据库自增ID 简单、有序 单点瓶颈、无法水平扩展 小规模系统
UUID 分布式、无中心 无序、索引效率低 对顺序无要求的场景
雪花算法 有序、高性能、分布式 依赖时钟、时钟回拨问题 大规模分布式系统
Redis生成 高性能、有序 依赖Redis 需要有序ID的场景
数据库分段 高可用、有序 实现复杂 大规模分布式系统

方案 说明 优缺点
抛出异常 检测到回拨直接抛出 简单，但会导致服务不可用
等待同步 等待时钟追上上次时间 可能造成性能抖动

1. 实现方案对比

Zookeeper分布式锁 可靠性高、可重入 性能较低 对可靠性要求高
数据库分布式锁 简单、无需额外组件 性能低、单点风险 小规模系统

总结

分布式系统的核心难题在于一致性和可靠性，掌握这些知识不仅能应对面试，还能设计出稳定的分布式系统。核心要点：

  1. 分布式事务：CAP/BASE理论、Seata AT/TCC/SAGA模式
  2. 分布式ID：雪花算法原理、时钟回拨解决方案
  3. 分布式锁：Redis锁实现、Redisson原理、Zookeeper锁对比
  4. 实战场景：跨服务一致性保证、幂等性设计

这些都是面试中高频问到的问题，一定要深入理解并能说出落地实现。
