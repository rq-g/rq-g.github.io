Spring事务底层实现——从@Transactional到数据库提交

前言

传播行为对比

传播行为 当前无事务 当前有事务 典型场景
REQUIRED 新建事务 加入当前事务 默认选择
REQUIRES_NEW 新建事务 挂起当前，新建事务 日志记录
NESTED 新建事务 嵌套事务(Savepoint) 子操作允许失败
SUPPORTS 非事务执行 加入当前事务 查询方法
NOT_SUPPORTED 非事务执行 挂起当前事务 不需要事务的操作
MANDATORY 抛异常 加入当前事务 强制要求事务
NEVER 非事务执行 抛异常 强制不要事务

面试高频问题

1. Spring事务的本质？

connection.setAutoCommit(false) + ThreadLocal绑定Connection到当前线程。

2. REQUIRED和REQUIRES_NEW的区别？

REQUIRED加入当前事务，一荣俱荣一损俱损；REQUIRES_NEW新建独立事务，互不影响。

3. NESTED和REQUIRES_NEW的区别？

NESTED是外层事务的子事务（Savepoint），外层回滚子事务也回滚；REQUIRES_NEW是完全独立的新事务，互不影响。

4. @Transactional(readOnly=true)的作用？

  1. 数据库层面：InnoDB不做change buffer，查询性能提升
  2. Spring层面：标记只读，部分事务管理器可以做优化
  3. 注意：readOnly只对事务内的查询有效

最佳实践

  1. 永远指定rollbackFor = Exception.class：避免checked异常不回滚
  2. 事务方法必须是public：private/protected事务不生效
  3. 避免同类调用：用AopContext或注入自身
  4. 事务粒度要小：只包裹必要的数据库操作，不要包含RPC调用
  5. 只读查询加readOnly：性能优化
  6. 日志操作用REQUIRES_NEW：避免日志回滚丢失

总结

Spring事务的本质是AOP + Connection.setAutoCommit(false) + ThreadLocal。2023年深入源码后，我彻底理解了事务传播行为的实现机制，也掌握了事务失效的所有场景。线上遇到数据不一致问题，不再盲目排查，而是从代理是否生效、异常是否被吞、传播行为是否正确三个维度系统分析。
