发布于 2023-04-16 分类：技术, 2023, 并发编程 👁

前言

线上一次OOM事故：同事在代码里new FixedThreadPool(100)处理大批量任务，导致线程数飙升、内存溢出。排查后发现是用了无界队列，任务堆积导致内存耗尽。这次事故让我意识到：不理解线程池核心参数，就是在给线上埋雷。

参数详解

参数 说明 建议
corePoolSize 核心线程数，即使空闲也不回收 CPU密集型：CPU核数+1；IO密集型：CPU核数×2
maximumPoolSize 最大线程数 根据业务压测确定
keepAliveTime 非核心线程空闲存活时间 一般60秒
workQueue 任务队列 必须有界！
threadFactory 线程工厂 自定义线程名
handler 拒绝策略 根据业务选择

面试高频问题

1. 线程池的核心参数有哪些？

corePoolSize、maximumPoolSize、keepAliveTime、workQueue、threadFactory、handler。核心线程数控制常驻线程，最大线程数控制峰值，队列缓冲溢出任务。

2. 提交任务的执行顺序？

核心线程 → 队列 → 非核心线程 → 拒绝策略。不是先创建最大线程数再放队列！

3. 如何设置线程池大小？

4. FixedThreadPool为什么会导致OOM？

内部使用LinkedBlockingQueue（无界队列），任务堆积时队列无限增长，最终OOM。

最佳实践

  1. 禁止使用Executors创建线程池：手动创建，明确参数
  2. 队列必须有界：防止OOM
  3. 自定义线程名：排查问题一目了然
  4. 选择合适的拒绝策略：重要任务不能丢弃
  5. 监控线程池状态：队列使用率、活跃线程数
  6. 优雅关闭：shutdown + awaitTermination
  7. 线程池隔离：不同业务使用不同线程池
  8. 动态调整：根据负载动态调整参数
  9. 指标监控：接入Prometheus监控告警
  10. 异常处理：统一处理任务异常

总结

线程池是并发编程的基础设施。2023年最大的认知转变是：线程池不是创建就完事，参数配置、拒绝策略、监控告警、优雅关闭，每一个环节都关乎线上稳定性。禁止使用Executors、队列必须有界、自定义线程名——这三条铁律必须遵守。

深入理解线程池的工作原理，结合业务场景合理配置参数，才能在高并发场景下保证系统的稳定性和性能。
