Redis持久化——RDB与AOF的选择

发布于 2023-03-26 分类：技术, 2023, Redis 👁

前言

线上Redis重启后数据全丢了，排查发现是持久化没配置好。当时只知道Redis有RDB和AOF两种持久化方式，但不清楚它们的区别和适用场景。2023年深入理解后，终于能根据业务选择合适的持久化方案。

场景 推荐方案 原因

面试高频问题

1. RDB和AOF的区别？

RDB是快照，定时保存，恢复快但可能丢数据；AOF是追加日志，实时记录，数据安全但恢复慢。推荐混合持久化。

2. AOF重写的原理？

fork子进程，根据当前内存数据生成最少的写命令。重写期间的新命令同时写入旧AOF和新AOF缓冲区，重写完成后合并替换。

3. Redis宕机数据能恢复多少？

RDB：丢失最后一次快照后的所有数据。AOF(everysec)：最多丢1秒数据。AOF(always)：基本不丢。

最佳实践

  1. 推荐混合持久化：Redis 4.0+默认开启
  2. AOF用everysec：安全与性能的平衡
  3. 控制实例内存：<10GB避免fork耗时
  4. 定期备份RDB：容灾恢复
  5. 监控持久化状态：fork耗时、重写状态

总结

Redis持久化是数据安全的最后防线。2023年深入理解后，核心认知是：RDB恢复快但可能丢数据，AOF数据安全但恢复慢，混合持久化兼顾两者。生产环境推荐AOF(everysec) + RDB备份 + 混合持久化。
